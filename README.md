# HomeGate

Конфигурация домашнего сервера: Home Assistant, MCP-гейт для Claude,
векторная память, мониторинг с алертами в Telegram, реверс-прокси.

Сервер стоит в частном доме и обслуживает домашнюю автоматику.
Хозяин дома пользуется им через своего Claude, администратор — по SSH.

---

## Что внутри

```
homeassistant/           Home Assistant (compose + configuration.yaml)
homegate/app/            MCP-гейт: FastAPI, типизированные инструменты
homegate/config/         config.example.json — шаблон (реальный конфиг не в git)
qdrant/                  векторная память (compose)
monitoring/              Prometheus, Grafana, Loki, Promtail, Alertmanager
dashboards/index.html    стартовая страница с состоянием дома
nginx/homegate.conf      реверс-прокси, TLS, ограничения доступа
nginx/ha.conf            Home Assistant на отдельном порту 8443
systemd/                 юниты homegate и node_exporter
docs/README.server.md    эксплуатационная документация машины
```

### Точки входа

| Адрес | Что |
|---|---|
| `https://<host>/` | стартовая страница: состояние дома, графики |
| `https://<host>/grafana/` | Grafana |
| `https://<host>/claude-mcp` | MCP-гейт |
| `https://<host>:8443/` | Home Assistant |

**Home Assistant не умеет работать в подкаталоге** — у него нет аналога
`serve_from_sub_path` из Grafana. Через `/ha/` он редиректит на
`/onboarding.html` без префикса и получается 404. Поэтому вынесен на
отдельный порт 8443, а `/ha/` отдаёт 302. При появлении домена правильнее
сделать поддомен `ha.<домен>` на 443.

## Ограничения доступа

В `location /claude-mcp` — список адресов, как на остальных серверах
контура:

```
allow 160.79.106.0/24;   # подсеть Anthropic, откуда приходят агенты Claude
allow <ноды администратора>;
allow 127.0.0.1;
allow 192.168.0.0/16;    # домашняя сеть
deny all;
```

Смысл списка — дать агентам Claude доступ к гейту. Токен при этом
остаётся обязательным, это второй рубеж. Закрывать гейт только домашней
сетью нельзя: агенты приходят снаружи и получат отказ.

Служебные эндпоинты `/api/home-states` и `/api/home-history/` ограничены
домашней сетью — они нужны браузеру, агентам туда не надо.

## MCP-гейт: почему не голый shell

Гейт даёт Claude доступ к дому. Ключевое решение — **инструменты
типизированные, а не произвольные shell-команды**.

Причина: дома цена ошибки выше, чем на рабочем сервере. Там упадёт сайт,
здесь останется без отопления живой человек, и чинить некому.

1. **Типизированные инструменты.** При `run_command` модель сама решает,
   какой командой добиться цели, и может ошибиться как угодно.
   При `device_control(entity, action)` максимум — не та лампа.

2. **Чтение свободно, запись по белому списку.** Разобраться в проблеме
   можно без спроса. Менять состояние дома — только для сущностей из
   `homeassistant.write_whitelist`.

3. **Домены `lock`, `climate`, `water_heater`, `valve`,
   `alarm_control_panel` запрещены жёстко** — в коде, не в конфиге.
   Даже если внести их в whitelist, `device_control` откажет. Замки,
   отопление, вода, щит — только руками хозяина.

4. **Роли разделены токенами.** У пользователя нет shell физически:
   `tools/list` его не отдаёт, прямой вызов возвращает отказ.

5. **Аудит.** Каждый вызов shell и каждое управление устройством пишется
   в `audit.log` с именем вызывающего.

### Инструменты

Роль `user` — 7 штук:
`memory_save`, `memory_search`, `memory_list`, `home_state`,
`sensor_history`, `home_anomalies`, `device_control`

Роль `admin` — те же плюс `run_command`, `service_status`.

`home_anomalies` — не «покажи датчики», а «что выглядит не так»:
молчащие больше суток сенсоры, севшие батареи, недоступные устройства.
Служебные сущности HA отфильтрованы, недоступные группируются по
устройству — иначе одно отключённое устройство даёт восемь строк.

### Векторная память

Qdrant плюс модель `intfloat/multilingual-e5-small` (384 измерения,
русский, локально на CPU). Модель грузится лениво при первом обращении:
старт гейта не ждёт, первый запрос после перезапуска ~15 секунд, дальше
0.02с на запись.

**E5 требует префиксов**: `query:` для поискового запроса, `passage:`
для сохраняемого текста. Без них качество заметно падает — это
особенность обучения модели.

В памяти лежит то, что полезно **хозяину дома**: что где стоит, какие
решения приняты и почему, что нельзя трогать, что не работает и по какой
причине. Хронология работ туда не пишется — для неё есть история git.

## Развёртывание с нуля

### 1. База

```bash
dnf install -y epel-release
dnf install -y git vim tmux htop bind-utils wget policycoreutils-python-utils
# Docker CE по инструкции docker.com, затем:
systemctl enable --now docker
```

### 2. Секреты

```bash
cp homegate/config/config.example.json /opt/homegate/config/config.json
cp monitoring/.env.example /opt/monitoring/.env
cp qdrant/.env.example /opt/qdrant/.env
cp monitoring/alertmanager/alertmanager.example.yml /opt/monitoring/alertmanager/alertmanager.yml
chmod 600 /opt/homegate/config/config.json /opt/monitoring/.env /opt/qdrant/.env
```

Токены генерировать на месте: `openssl rand -hex 32`.
Не переиспользовать токены из других контуров.

### 3. Сертификат

```bash
mkdir -p /etc/nginx/ssl
openssl req -x509 -nodes -newkey rsa:2048 -days 3650 \
  -keyout /etc/nginx/ssl/selfsigned.key -out /etc/nginx/ssl/selfsigned.crt \
  -subj "/CN=homegate.local"
chmod 600 /etc/nginx/ssl/selfsigned.key
```

При появлении домена: `certbot --nginx -d <домен>`.

### 4. Сервисы

```bash
cd /opt/qdrant && docker compose up -d
cd /opt/monitoring && docker compose up -d
cd /opt/homeassistant && docker compose up -d
systemctl enable --now node_exporter homegate nginx
```

### 5. Home Assistant

Первичная настройка только через браузер: `https://<host>:8443/`,
создать аккаунт владельца. Интеграции добавляются там же — через
командную строку config flow не пройти.

Для метрик нужен long-lived token: профиль → Безопасность → Токены
долгосрочного доступа. Положить в `/opt/monitoring/prometheus/ha_token`
и в `homeassistant.token` конфига гейта.

Для стартовой страницы — тот же токен в
`/etc/nginx/secrets/ha_auth.conf` в виде
`proxy_set_header Authorization "Bearer <токен>";`

## Интеграции

### Tuya / Smart Life

**Схема авторизации изменилась.** С версии HA 2024.x интеграция не
спрашивает Access ID и Secret — она просит **User Code** из приложения:
*Smart Life → Я → Настройки → Учётная запись и безопасность → Код
пользователя*. Access ID туда не подходит, будет `USERCODE_INCORRECT`.

Аккаунт разработчика на `iot.tuya.com` всё равно нужен:

- Cloud Project с типом **Smart Home**
- Data Center должен совпадать с регионом аккаунта Smart Life
  (для России обычно **Central Europe**). Ошибка здесь = устройства
  просто не появятся
- Service API: минимум `IoT Core`, `Authorization Token Management`,
  `Smart Home Scene Linkage`
- Devices → Link Tuya App Account → QR-код → отсканировать в Smart Life

**Аккаунт бесплатный, но требует продления вручную** примерно раз в
полгода кнопкой *Extend Trial Period*. Если забыть — устройства
отвалятся до следующего продления. Стоит поставить напоминание.

Веб-консоль Tuya из России может грузиться криво (иконки есть,
инструментов нет), но API `openapi.tuyaeu.com` при этом доступен.

### Что не работает через облако Tuya

ИК-передатчики (`Smart IR`, кондиционер, телевизор) приходят как
`unsupported` — облачный API не отдаёт ИК-команды. Управлять ими можно
только из родного приложения. Нужен локальный путь или отдельная
интеграция.

### Bluetooth

Встроенный адаптер подхватывается HA автоматически. Нужны `NET_ADMIN` +
`NET_RAW` в compose и проброс `/run/dbus`, иначе `PermissionError`.

**Tuya BLE-устройства локально читаются не все.** Интеграция
`ha_tuya_ble` требует ключ шифрования из облака, а список поддерживаемых
`product_id` невелик. Устройство может отлично ловиться по эфиру, но
нагрузка останется зашифрованной. Проверять по списку в README
интеграции до того, как рассчитывать на локальную работу.

#### Разобранный случай: датчик температуры и влажности

Датчик `TyOS` (`product_id vlzqwckk`) — наглядный пример того, чем
облачная привязка отличается от локального чтения. Разбор занял день,
поэтому выводы стоит держать под рукой.

*Привязка через облако работает.* После добавления в Smart Life датчик
приезжает в HA обычной облачной интеграцией: температура, влажность,
заряд батареи. Ничего допиливать не нужно.

*Но у датчика нет своего шлюза.* В реестре устройств у него
`via_device_id: None`. В облако он ходит через телефон с открытым
приложением — телефон и есть ретранслятор. Выключите Bluetooth на
телефоне, и показания замрут: проверено наблюдением в двадцать минут,
значение не сдвинулось, а приложение само сообщило, что связи нет.
После включения Bluetooth данные пришли, но потребовался ручной
перезапуск приложения.

Практический смысл: **пока хозяин дома — показания живые, ушёл —
замерли.** Для дежурного мониторинга дома это не годится.

*Перепривязать датчик напрямую к серверу нельзя.* Это не ограничение
железа, а свойство протокола. Tuya BLE — не обычный BLE-датчик, к
которому можно подключиться и почитать характеристики: он вещает
зашифрованный поток и слушает только того, кто знает `local_key`.
Привязка происходит внутри экосистемы Tuya, а не спариванием на уровне
ОС. Второго хозяина устройство иметь не умеет.

*Ключ через штатную интеграцию не достать.* Токен, который HA получает
при облачной авторизации, выдан под узкий набор путей и `local_key` не
отдаёт. Рабочие пути API (`tuya_sharing`, модуль `device.py`):

```
/v1.0/m/life/users/homes          — список домов, работает
/v1.0/m/life/ha/home/devices      — устройства дома, permission deny
/v1.0/m/life/ha/devices/detail    — карточка устройства, permission deny
```

Коды ошибок, которые встретятся по дороге: `1108 uri path invalid` —
неверный путь; `1013 request time is invalid` — запрос без подписи, одним
`access_token` не обойтись; `1106 permission deny` — путь и подпись верны,
но у токена нет прав.

*Что осталось теоретически.* Интеграция `ha_tuya_ble` получает ключ
через собственную авторизацию в облаке, а не через токен HA, и читает
эфир локально — телефон становится не нужен. Упирается в то, что
`product_id` конкретной модели может не поддерживаться ни одним форком,
и разбор полей придётся писать вслепую.

*Вывод.* Если нужна температура, которая не зависит от телефона, дешевле
взять Zigbee-датчик и подключить к своему стику: без облака, без
телефона, без шифрования. Возня с Tuya BLE оправдана только когда
устройство уже куплено и заменить его нечем.

## Мониторинг

### Метрики

Prometheus собирает четыре цели: сам себя, `node_exporter`,
Home Assistant и Docker.

Эндпоинт `/api/prometheus` в HA **по умолчанию выключен** — без секции
`prometheus:` в `configuration.yaml` отдаёт 404.

Метрики дома приходят с префиксом `hass_`: `hass_sensor_power_w`,
`hass_sensor_energy_kwh`, `hass_binary_sensor_state`, `hass_cover_state`
и другие.

**cAdvisor не работает с Docker 29** — версии 0.49 и 0.52 ищут слои в
`/var/lib/docker/image/overlayfs`, которого там нет, и отдают только
метрики хоста без разбивки по контейнерам. Вместо него включён
встроенный экспортёр Docker: в `/etc/docker/daemon.json` прописан
`"metrics-addr": "172.17.0.1:9323"`. Именно адрес docker0, не
`127.0.0.1` — иначе контейнер Prometheus не достучится. Нужен полный
`systemctl restart docker`, reload не подхватывает.

### Логи

Loki и Promtail собирают логи nginx, гейта, аудита, Home Assistant и
системный журнал. Loki подключён к Grafana как источник данных.

Логи гейта и HA связаны симлинками в `/var/log/homegate/` и `/var/log/ha/`,
чтобы Promtail видел единый каталог.

**Журнал systemd должен быть постоянным.** По умолчанию `Storage=auto`
означает, что журнал живёт в `/run/log/journal` и теряется при
перезагрузке, а Promtail не может к нему подключиться. Нужно
`Storage=persistent` в `/etc/systemd/journald.conf` и создать
`/var/log/journal`.

### Дашборды

- **Node Exporter Full** — 31 панель по хосту
- **Дом** — потребление, устройства на связи, свет, розетки, накопленный
  расход, напряжение, временные шкалы состояний ворот, света и датчика дыма

### Алерты

Семь правил в `monitoring/prometheus/rules/home.yml`: пожар, ворота
открыты дольше 15 минут, напряжение вне 200–245 В, HA не отвечает, диск
заполнен, перегрев процессора, мало памяти. Пожар отправляется без
задержки и повторяется каждые полчаса.

Уведомления идут в Telegram. **В России API Telegram недоступен
напрямую** — в `alertmanager.yml` указан `telegram_api_url`, ведущий на
прокси. Токен бота лежит в отдельном файле `bot_token`.

Файлу с токеном нужен SELinux-контекст `container_file_t` и права на
чтение: контейнер Alertmanager работает под `nobody` и файл с `600`
прочитать не может.

Правила подхватываются не сразу, если файл создан после старта
Prometheus — помогает `curl -X POST http://127.0.0.1:9090/-/reload`.

## Стартовая страница

`https://<host>/` показывает живое состояние дома: датчик дыма, ворота,
свет, розетки с потреблением, список устройств не на связи. Плюс графики
за сутки — напряжение в сети и потребление розеток. Обновляется раз в
30 секунд, графики раз в 5 минут.

**Токен HA в браузер не попадает.** Страница ходит на свой же сервер
(`/api/home-states` и `/api/home-history/`), nginx подставляет заголовок
из `/etc/nginx/secrets/ha_auth.conf` и проксирует в HA. Эндпоинты только
на чтение и ограничены домашней сетью.

Графики рисуются в SVG вручную — тянуть библиотеку ради двух линий на
слабом процессоре незачем.

Два момента про данные:

- **HA пишет только изменения.** Если розетки выключены, у мощности будет
  одна точка за сутки, и график покажет «мало точек». Это не ошибка, он
  оживёт при нагрузке.
- При усреднении напряжения датчики, ещё не давшие первого значения, не
  учитываются. Иначе ноль занижает среднее и в начале графика появляется
  провал.

## Грабли

- **SELinux Enforcing.** Новые каталоги под логи требуют контекста,
  иначе systemd падает с `status=209/STDOUT`:
  `semanage fcontext -a -t var_log_t "/path(/.*)?" && restorecon -Rv /path`
  Для бинд-маунтов в Docker — суффикс `:Z`. Файлам, которые читает
  контейнер, — `chcon -t container_file_t`.

- **nginx 1.20**, не 1.25: `listen 443 ssl http2;` одной строкой,
  отдельная директива `http2 on;` не поддерживается.

- **Grafana в подкаталоге** требует и `GF_SERVER_ROOT_URL`, и
  `serve_from_sub_path`, и `proxy_pass` **без** трейлинг-слеша. Слеш
  срезает префикс, получается цикл редиректов.

- **torch тянет CUDA.** Обычный `pip install sentence-transformers`
  ставит библиотеки NVIDIA и раздувает venv с 40 МБ до 5 ГБ на машине
  без видеокарты. Правильно:
  `pip install torch --index-url https://download.pytorch.org/whl/cpu`

- **CH340 (Zigbee-стики)** даёт нестабильные имена устройств:
  `ttyUSB0` может стать `ttyUSB1`. Привязывать по
  `/dev/serial/by-id/`, иначе после перезагрузки адаптер потеряется.

- **Правка docker-compose скриптом** — источник поломок. Надёжнее
  пересобрать файл целиком через heredoc, чем патчить заменами.

## Безопасность

- Секретов в репозитории нет и быть не должно. `.gitignore` блокирует
  `config.json`, `*.env`, `*.key`, `*.crt`, `*_token`, `ha_token`,
  `bot_token`, `secrets/`, `models/`, логи, venv, storage
- Токены генерировать на месте, не переиспользовать между контурами
- Датчики дыма и CO **не вносить в `write_whitelist`**. Пожарная
  сигнализация должна работать автономно; HA — дополнительный слой
  уведомлений, не замена
- Перед коммитом проверять:
  `git diff --cached | grep -iE "token|secret|password"`

## Что не сделано
